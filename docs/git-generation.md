# Git generation, safety, and resume

`apply` is the only command that materializes commits. It consumes a frozen,
checksummed plan and never derives new commit choices while it runs.

## Review workflow

Always inspect and dry-run first:

```bash
gm plan inspect ./art/plans/latest.json
gm apply ./art/plans/latest.json --dry-run
```

Dry-run verifies Git availability, repository type, cleanliness, branch,
remotes, base commit, plan application state, and all required authorization
flags. It does not initialize a repository or create a commit.

For a new target:

```bash
gm apply ./art/plans/latest.json --init-repository
```

For an existing target:

```bash
gm apply ./art/plans/latest.json --allow-existing-repository
```

Add `--allow-repository-with-remotes` whenever any remote is configured. The
flag acknowledges risk; it does not authorize or trigger a push.

Without `--yes`, an interactive apply prints the absolute target path and asks
you to type the 16-character plan ID. A non-interactive process must use
`--yes`. Treat it as a high-impact flag and only add it after a successful
dry-run has been reviewed.

## New and existing repository invariants

A `new` plan may target a missing path, an empty directory, or a repository with
no commits. A non-empty non-repository directory and a repository that already
has unrelated commits are rejected.

An `existing` plan requires:

- `--repository-mode existing` and `--expected-head HASH` at plan time;
- `--allow-existing-repository` at validation/application time;
- an existing Git repository with a clean worktree and index;
- the current branch to equal the planned branch;
- before the first generated commit, `HEAD` to equal `expectedHead` exactly;
- while resuming, the planned base to remain an ancestor of `HEAD`.

Detached HEADs and branch mismatches are rejected. The executor appends commits;
it does not reset, amend, rebase, force-push, or otherwise rewrite history.

## Commit modes

The default `empty` mode uses `git commit --allow-empty`; it does not add an
artificial content file.

File mode appends one tab-separated line per commit to a safe repository-relative
path and stages it:

```bash
gm plan \
  --project ./art --repo ../history \
  --author-name "Example User" --author-email user@example.com \
  --commit-mode file --file-path .git-mosaic/activity.log
```

Absolute paths and paths containing a `..` segment are rejected. File mode
changes repository content and should be chosen only when that artifact is
wanted; empty mode is preferable for pure artwork.

## Reproducible commit metadata

For every commit, the executor sets author and committer name/email/date from the
plan. Unless separately supplied, committer identity equals author identity.
Dates begin at local noon and advance one second per commit on that day.

The plan message supports these placeholders:

- `{date}` and `{timestamp}`
- `{index}` and `{total}` (within the day)
- `{intensity}`
- `{project}`

Each commit also receives machine-readable trailers:

```text
Git-Mosaic-Plan: 0123456789abcdef
Git-Mosaic-Step: 42/380
Git-Mosaic-Date: 2025-06-14
```

Local hooks, configured commit signing, and terminal prompting are disabled for
generated commits. This avoids machine-specific hook/signing behavior and means
repository policies normally enforced only by local hooks are not run. Review
the plan and repository policy yourself before applying.

## Interruption and resume

On `SIGINT`, the CLI stops between commits, verifies the partial history, and
writes `<plan>.apply-report.json`. Running the same apply command again scans
commit trailers, verifies the exact contiguous prefix, and continues with the
next step. Already-created steps are not duplicated.

The detected states are:

- `not_started`: no matching plan trailers;
- `partial`: a valid contiguous prefix exists;
- `complete`: all steps exist (reapplication is rejected);
- `divergent`: duplicate, missing, malformed, or mismatched steps exist
  (application is rejected).

Do not edit generated messages/trailers or reorder/squash generated commits
before resuming. An abrupt process or machine failure can leave file mode with a
dirty appended file before its commit; the executor intentionally refuses to
continue until the repository is inspected and restored to a known clean state.
It does not discard changes automatically.

## Publishing and disclosure

No code path runs `git push`. If you decide to publish, configure a remote and
push manually after inspecting `git log`, the branch, and repository contents.
Do not force-push over shared history.

The generated timestamps and density are artwork rather than evidence of work.
Include a disclosure in the repository README, for example:

> This repository's commit history was generated by git-mosaic as contribution
> artwork. It does not represent the timing or volume of software development.
